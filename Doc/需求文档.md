# 需求文档

> 版本：v2.0 | 日期：2026-07-15

本文档描述**星尘（Stardust）分布式服务平台**的愿景、核心目标和功能需求。完成状态在[功能清单](功能清单.md)中追踪，详细设计在各模块架构文档中展开。

---

## 1. 背景与愿景

### 1.1 系统定位

星尘是一个**轻量级分布式服务框架**，提供集群管理、配置中心、性能监控（APM）、远程发布、服务治理等一站式能力。应用服务犹如宇宙中的尘埃，计算机节点是为星星，星星与尘埃构成浩瀚星空，即为星尘。

| 工程 | 定位 | 部署方式 | 说明 |
|------|------|----------|------|
| **StarServer** (Stardust.Server) | 服务平台后端。API / 配置 / 注册 / 发布 / 指标聚合 / 统计计算 | 中心部署 1~N 节点 | 可横向扩展 |
| **StarWeb** (Stardust.Web) | 管理与可视化。基于 NewLife.Cube 的后台管理系统 | 中心部署 1 节点 | 可与 StarServer 同机 |
| **StarAgent** | 节点代理。进程守护、远程发布、监控采集 | 每台服务器一实例 | Windows 服务 / Linux systemd |
| **StarGateway** | 流量网关。反向代理、动态路由、负载均衡 | 按需部署 | 可选组件 |
| **Stardust（核心库）** | 客户端 SDK。配置拉取、注册发现、调用追踪、性能和日志上报 | NuGet 包引用 | 与业务应用同进程 |
| **Stardust.Extensions** | ASP.NET Core 中间件扩展 | NuGet 包引用 | 便捷接入 |
| **Stardust.Data** | 数据层。XCode ORM 实体模型与统计任务 | 与 StarServer 同进程 | 多数据库兼容 |
| **DeployAgent** | 部署代理。构建与分发支撑 | 按需部署 | 可选组件（编译节点） |

### 1.2 愿景

星尘做成了会是什么样子：

1. **一键接入分布式**：几行代码完成分布式接入，应用零改造即可获得配置中心、注册中心、APM 能力
2. **超轻量无侵入**：核心库零第三方重量级依赖，单进程即可运行，自动建库建表，无需额外基础设施
3. **全平台覆盖**：同一生态覆盖 .NET Framework 4.5 起至最新 .NET 10，Windows/Linux/ARM/IoT 全平台
4. **自带运维体系**：StarAgent 守护进程 + 远程发布 + 指标&日志采集，极大降低分布式运维复杂度
5. **一体化管理**：节点管理、配置中心、注册中心、APM、日志中心、远程发布统一模型与界面

| 多语言 SDK 完善 | SDK 目录已有 Go/Python/JS 示例但未完整维护 | 社区贡献或商务需求 |
| 网关 Web UI 配置 | 网关基础能力稳定后可视化管理 | StarGateway 基础能力完善 |
| 限流/熔断 | 需要定义配置模型和算法实现 | 负载均衡和健康检查完善后 |
| HTTP/2 / HTTP/3 支持 | 当前 HTTP/1.1 满足 MVP 需求 | 出现必须 HTTP/2 的后端或客户端需求 |

## 5. 不做什么

- **不替代专业日志分析平台**（如 ELK）。日志中心定位为应用运行日志集中查看，不提供全文索引等高级能力
- **不替代专业 APM 系统**（如 SkyWalking/Pinpoint）。采用客户端采样统计模式，非全量采集
- **不做 SaaS 化多租户**。定位为企业内部部署
- **不强制依赖外部中间件**。核心链路只需数据库即可运行，Redis/消息队列为可选增强
